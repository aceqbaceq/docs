создавался tcp конект (сокет) между процессами
через него клиент слал запрос на сервер  в виде текста (отформатиованнго по формату http). потом ждал ответ в это время ничего нового
по этому tcp конекту делать нельзя. сервер отправлял ответ и (!важно) сам тут же инциировал закрытие тцп сокета. клиент получал ответ.

что дал http 1.1
он дал то что 
  *) когда мы шлем запрос через тцп сокет на сервер то мы передаем текст параметр keep-alive
     это дает то что когда сервер отправлял ответ он не закрывал тцп сокет. поэтому клиент
     мог отправить новый запрос через этот же тцп сокет

  *) по идее клиент мог отправить сразу несколко http запросов один за другм на сервер
     что как бы плюс, но сервер в ответ слал ответы строго по одному . тоесть если сревер
     щас готовит ответ на зпрос номер два но при этом он мог бы уже отправить запрос номер
     три но нихуя. нужно подждодать кода будет готов ответ номер два. послать его и олько потом
     можно отослать ответ номер три

   *) из этого вытекает что если мы хотим чтобы страница веб сайта загрузилась побыстрее
      то клиент должен по прежнему создавать дохрена тцп конектов к серверу для получения 
      кусков страницы в паралельном режиме для ускорения загрузки. условно говоря на странице
      20 картинок каждая имеет свой url. значит надо создать 20 тцп конектов чтобы их загружать
      одноврменно



что дал http 2

  *) до этого запросы http были в ткстовом виде по крайней мере заголовки. пейлоад мог быть
